Сначала не пиши код. Сначала изучи структуру проекта, существующие модели, urls, views, templates и базу данных. После этого предложи архитектуру доработки и только потом переходи к коду.

У пользователей уже реализованы календарь и статистика. Нужно доработать проект и добавить полноценный личный кабинет, систему чатов, оповещения сотрудников и тикеты для вопросов.

Нужно продумать и реализовать функционал со стороны администратора и со стороны учителей.

1. Личный кабинет пользователя

Для каждого пользователя нужно сделать личный кабинет.

В личном кабинете учитель должен видеть:
- свои личные данные;
- календарь;
- статистику;
- уведомления;
- сообщения;
- тикеты/вопросы, которые он создавал.

Администратор в личном кабинете должен видеть:
- календарь;
- статистику;
- список пользователей;
- сообщения;
- тикеты;
- архив тикетов;
- управление уведомлениями сотрудников.

2. Система чатов

Нужно реализовать чат между пользователями.

Функционал:
- пользователь может выбрать другого пользователя из списка;
- пользователь может написать сообщение;
- сообщения должны сохраняться в базе данных;
- у каждого пользователя должен быть список диалогов;
- непрочитанные сообщения должны визуально отличаться;
- после прочтения сообщение должно менять статус.

Нужно предусмотреть модель сообщений, например:
- отправитель;
- получатель;
- текст сообщения;
- дата и время отправки;
- статус прочитано/не прочитано;
- тип сообщения.

3. Системный аккаунт для оповещений

Нужно создать специальный системный аккаунт:

Название: «Система оповещения»

Особенности:
- этот аккаунт может отправлять сообщения пользователям;
- пользователи не могут отвечать этому аккаунту напрямую;
- системные сообщения создаются автоматически администратором через календарь;
- такие сообщения должны отображаться в списке сообщений пользователя.

4. Кнопка «Оповещение сотрудников» в календаре администратора

В календаре администратора нужно добавить кнопку:

«Оповещение сотрудников»

Логика:
- администратор выбирает конкретный день в календаре;
- нажимает кнопку «Оповещение сотрудников»;
- система смотрит, какие сотрудники выбраны/назначены на этот день;
- каждому выбранному сотруднику приходит сообщение от аккаунта «Система оповещения».

В сообщении должен быть список замен для этого сотрудника:

Пример текста сообщения:

Здравствуйте!

Вам назначены замены на [дата]:

1. [Время] — [номер урока] урок — [класс]
2. [Время] — [номер урока] урок — [класс]
3. [Время] — [номер урока] урок — [класс]

Пожалуйста, подтвердите ознакомление или задайте вопрос.

5. Кнопки в системном сообщении

В системном сообщении должны быть две кнопки:

- «Ознакомлен»
- «Есть вопросы»

Изначально системное сообщение должно быть серым.

Если сообщение прочитано, но пользователь еще не нажал кнопку, оно становится белым.

Если пользователь нажал «Ознакомлен»:
- сообщение получает статус подтверждено;
- сообщение подсвечивается зеленым цветом;
- администратор должен видеть, что пользователь ознакомился.

Если пользователь нажал «Есть вопросы»:
- пользователя нужно перенаправить в чат «Вопросы»;
- автоматически создается новый тикет;
- тикет связан с этим системным сообщением;
- доступ к тикету имеют только:
  - пользователь, который создал тикет;
  - администраторы.

6. Система тикетов «Вопросы»

Нужно реализовать отдельный раздел тикетов.

Тикет создается, когда пользователь нажимает кнопку «Есть вопросы» в системном уведомлении.

У тикета должны быть:
- автор;
- связанное системное сообщение;
- тема;
- сообщения внутри тикета;
- статус: открыт / закрыт;
- отметка важности;
- дата создания;
- дата закрытия;
- список участников.

Учитель должен иметь возможность:
- открыть тикет;
- написать вопрос;
- видеть ответы администраторов;
- видеть статус тикета;
- продолжать переписку, пока тикет открыт.

Администратор должен иметь возможность:
- видеть список всех открытых тикетов;
- отвечать в тикетах;
- закрывать тикет, если вопрос решен;
- отметить тикет важным;
- снять отметку важности;
- видеть архив закрытых тикетов.

7. Важные тикеты

У администратора должна быть кнопка:

«Отметить важным»

Если тикет отмечен важным:
- он подсвечивается в списке тикетов;
- рядом с ним отображается звездочка;
- важные тикеты должны быть выше обычных в списке.

8. Закрытие тикета

У администратора должна быть кнопка:

«Закрыть тикет»

После закрытия:
- тикет нельзя продолжать обычному пользователю;
- тикет переносится в архив;
- в архиве его можно просмотреть;
- история сообщений сохраняется.

Желательно предусмотреть возможность:
- открыть закрытый тикет повторно, если администратор решит это сделать.

9. Архив тикетов

Нужно сделать отдельный раздел:

«Архив тикетов»

В архиве администратор должен видеть:
- закрытые тикеты;
- автора тикета;
- дату создания;
- дату закрытия;
- связанное уведомление;
- историю сообщений;
- был ли тикет важным.

Учитель должен видеть в архиве только свои закрытые тикеты.

10. Права доступа

Нужно строго разграничить доступ.

Учитель:
- видит только свои сообщения;
- видит только свои уведомления;
- видит только свои тикеты;
- может писать другим пользователям;
- может создавать тикет через кнопку «Есть вопросы»;
- не может писать аккаунту «Система оповещения»;
- не может видеть чужие тикеты.

Администратор:
- видит все тикеты;
- может отвечать в тикетах;
- может закрывать тикеты;
- может отмечать тикеты важными;
- может отправлять системные оповещения сотрудникам через календарь;
- может видеть статусы ознакомления сотрудников.

Системный аккаунт:
- используется только для автоматических уведомлений;
- не принимает ответы;
- не отображается как обычный пользователь для выбора в чате.

11. Статусы сообщений

Для системных уведомлений нужны статусы:

- created — создано, еще не прочитано;
- read — прочитано;
- acknowledged — пользователь нажал «Ознакомлен»;
- question — пользователь нажал «Есть вопросы» и создал тикет.

Цвета:
- серый — новое системное уведомление;
- белый — прочитано, но действие не выбрано;
- зеленый — пользователь нажал «Ознакомлен».

12. Что нужно реализовать технически

Нужно:
- проанализировать текущие модели пользователей, календаря и статистики;
- не сломать существующий функционал;
- добавить новые модели для:
  - личного кабинета;
  - обычных сообщений;
  - системных уведомлений;
  - тикетов;
  - сообщений внутри тикетов;
  - статусов ознакомления;
- добавить views;
- добавить urls;
- добавить шаблоны;
- добавить формы;
- добавить права доступа;
- добавить миграции;
- связать новый функционал с уже существующим календарем;
- добавить удобный интерфейс для администратора и учителей.

13. Желательная структура моделей

Предложи и реализуй модели примерно такого типа:

- User или существующая модель пользователя;
- ChatMessage;
- SystemNotification;
- NotificationReplacementItem;
- NotificationStatus;
- Ticket;
- TicketMessage.

Нужно учитывать, что у одного системного уведомления может быть несколько строк замен:
- время;
- номер урока;
- класс;
- сотрудник;
- дата.

14. Интерфейс администратора

Для администратора нужно сделать страницы:

- список пользователей;
- календарь;
- кнопка «Оповещение сотрудников»;
- предварительный просмотр уведомления перед отправкой;
- список отправленных уведомлений;
- статусы ознакомления сотрудников;
- список открытых тикетов;
- важные тикеты;
- архив тикетов;
- чат.

15. Интерфейс учителя

Для учителя нужно сделать страницы:

- личный кабинет;
- календарь;
- статистика;
- сообщения;
- системные уведомления;
- кнопки «Ознакомлен» и «Есть вопросы»;
- мои тикеты;
- архив моих тикетов;
- чат с другими пользователями.

16. Поведение после отправки уведомления

Когда администратор отправляет уведомление:
- система создает системное сообщение для каждого выбранного сотрудника;
- сообщение появляется у сотрудника в личном кабинете и в сообщениях;
- администратор может видеть, кому уведомление отправлено;
- администратор может видеть, кто ознакомился;
- администратор может видеть, кто нажал «Есть вопросы».

17. Нужно также продумать UX

Важно сделать интерфейс понятным:

Для учителя:
- сразу видно новые уведомления;
- понятно, что нужно нажать «Ознакомлен» или «Есть вопросы»;
- после нажатия «Есть вопросы» сразу открывается тикет.

Для администратора:
- видно, кто не ознакомился;
- видно, у кого есть вопросы;
- важные тикеты выделяются;
- закрытые тикеты не мешают работе, но доступны в архиве.
